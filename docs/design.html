Architecture, components, and design decisions for research-agent v2

Architecture Overview

research-agent is a single-file Python CLI app (~1500 lines) organized as a pipeline of specialized modules, each responsible for one phase of the research workflow. The GEPA self-improvement loop wraps the entire pipeline to evolve prompt strategies across generations.

Component Breakdown

TaskPlanner

Decomposes the user's query into sub-questions and a structured research plan. Uses heuristic templates (not LLM) to stay fast and deterministic. Generates 3-5 sub-questions targeting different angles.

WebCrawler

Multi-source web retrieval layer with three backends:

BackendSourceStatus
DuckDuckGo HTMLGeneral web searchPrimary (may hit CAPTCHA)
Wikipedia APIWikipedia articles + searchFallback (rate-limited)
SearXNGSelf-hosted instanceOptional (--searx-instance)

Uses asyncio.Semaphore for concurrency control, URL dedup cache, and HTTPX async client.

ContradictionEngine

Analyzes multiple source texts for conflicting statistics, contradictory claims, and divergent narratives. Produces a contradiction summary with conflict lines and source attribution.

ReportSynthesizer

Generates a structured markdown report with executive summary, key findings, contradictions, and conclusions. Uses heuristic templates with LLM-grade scoring via the ModelGrader.

GEPA Loop

The Generative Evolutionary Prompt Adaptation loop is the core self-improvement mechanism:

  1. Generate — Create a pool of PromptCandidate variants by mutating prompt templates for each module
  2. Evaluate — Run each candidate through the full eval suite (3 tasks × 4 checks each)
  3. Grade — Score each candidate on accuracy (pass rate via heuristic grader) and speed
  4. Reflect — Analyze failed checks to produce structured Lesson instances
  5. Mutate — Apply lessons to generate the next generation (targeted mutations on failing modules)
  6. Select — Choose the winner via Pareto frontier (accuracy first, then speed)
  7. Self-Patch — Generate _self_patch.py to bake winning prompts into the source

Grading System

Two-tier grading:

GraderMethodUse Case
HeuristicGraderKeyword-based scoring on report structureDefault — fast, deterministic, always works
OllamaGraderLocal LLM inference via Ollama APIWhen Ollama available — richer quality signal

The heuristic grader checks: executive summary presence, finding structure, source citations, contradiction analysis, and conclusion completeness. 4 checks per task, 12 total across the eval suite.

Eval Suite

Built-in evaluation tasks that exercise different research scenarios:

TaskQueryChecks
Ambiguous Health ClaimBenefits of intermittent fasting4
Emerging Technology ImpactAI in climate change research4
Historical RevisionismFall of the Roman Empire4

Each task is run multiple times (--eval-runs N) for statistical significance. Pass rate and duration are reported per-task.

Design Decisions

Single file over package

Keeps deployment trivial: pip install git+... or just download the file. Self-patching is simpler without package imports. Downside: 1500 lines in one file, but the module structure (classes + clear sections) keeps it navigable.

Heuristic over LLM (by default)

LLM calls add latency (5-15s per report), require GPU memory, and introduce nondeterminism. The heuristic grader is instant, deterministic, and sufficient for regression detection. LLM grading is available as an upgrade.

Separate self-patch file

Rather than modifying the script in-place during GEPA, we generate _self_patch.py — a standalone script that applies learned prompt improvements. This avoids string-collision bugs during in-place rewriting and gives the user a chance to review changes before applying.

State persistence via JSON

Research progress is saved to JSON state files. This enables resumability (if a crawl is interrupted, it picks up where it left off) and debugging (inspect what was searched, fetched, and synthesized). Unique paths per run prevent cross-contamination.

Data Flow

1. Input Query
2. TaskPlanner: decompose → [SubQ1, SubQ2, ...]
3. WebCrawler:
   a. Try SearXNG (if configured)
   b. Try DDG HTML search
   c. Fallback to Wikipedia API
   d. Fetch page content (with Semaphore concurrency)
4. ContradictionEngine: analyze sources → conflict lines
5. ReportSynthesizer: synthesize → structured markdown
6. Report output (stdout + state file)
7. GradeReport output (eval mode → pass/fail per check)
